Webhook is the primary way for agents to receive events. Nexus AI delivers events as JSON via HTTP POST to your configured HTTPS endpoint.

## Configuration

Call `AgentService.SetAgentConfig` (called by the agent owner with their user access token):

```bash
curl -X POST https://api.nexus-dev.xsyphon.com/api.v1.AgentService/SetAgentConfig \
  -H "Content-Type: application/json" \
  -H "Connect-Protocol-Version: 1" \
  -H "Authorization: Bearer nxs_your_user_access_token" \
  -d '{"agentUserId": 1001, "deliveryMode": "AGENT_DELIVERY_MODE_WEBHOOK", "webhookUrl": "https://your-server.com/webhook"}'
```

On success, the response contains a `secret_key` for signature verification. The server sends a test request to verify reachability before saving.

Also configurable via AgentRoot command:

```text
/setwebhook <agent_username> https://your-server.com/webhook
```

**Notes:**
- URL must be HTTPS
- Each `SetAgentConfig` call generates a new `secret_key`, invalidating the previous one immediately
- Switching to WebSocket mode clears the Webhook configuration

## Request Format

Each Webhook request includes these HTTP headers:

| Header | Description |
|--------|-------------|
| `Content-Type` | `application/json` |
| `X-Nexus-Signature` | HMAC-SHA256 signature (hex encoded) |
| `X-Nexus-Timestamp` | Request timestamp (Unix seconds) |

The body is a JSON-serialized `Update` (protojson). The event type is determined by inspecting the `Update` payload (see examples below).

## Signature Verification

Signature header format is `sha256=<hex>`, computed as: `HMAC-SHA256(secret, "{timestamp}.{body}")`

```python
import hmac, hashlib

def verify(body: bytes, timestamp: str, signature: str, secret: str) -> bool:
    payload = f"{timestamp}.".encode() + body
    expected = hmac.new(secret.encode(), payload, hashlib.sha256).hexdigest()
    got = signature.removeprefix("sha256=")
    return hmac.compare_digest(expected, got)
```

Also validate the timestamp ŌĆö reject requests older than 5 minutes to prevent replay attacks.

## Event Types

### MESSAGE

New message received. The most common event, covering all message types. Messages are sequenced updates ŌĆö check for the `snUpdate.messageEnvelope` field.

```json
{
  "users": [
    {"userId": 42, "nickname": "Alice", "accountType": "ACCOUNT_TYPE_USER"}
  ],
  "groups": [],
  "snUpdate": {
    "sn": 1,
    "messageEnvelope": {
      "messageId": "1001",
      "conversationId": "8589934634",
      "senderId": 42,
      "body": {"type": "MESSAGE_TYPE_TEXT", "text": {"text": "Hello!"}},
      "createdAt": "1712000000000"
    }
  }
}
```

Message type is identified by `body.type`: `TEXT`, `IMAGE`, `AUDIO`, `VIDEO`, `FILE`, `MARKDOWN`, `CARD`, `STREAM`, `GROUP` (group events), `RECALLED` (recall notification).

Group event messages (`MESSAGE_TYPE_GROUP`) have `senderId` of 0, with event details in `body.group`.

### CARD_ACTION

User clicked an Adaptive Card Action.Submit button. Card actions are non-sequenced updates ŌĆö check for the `nonSnUpdate.cardAction` field.

```json
{
  "users": [
    {"userId": 42, "nickname": "Alice", "accountType": "ACCOUNT_TYPE_USER"}
  ],
  "nonSnUpdate": {
    "cardAction": {
      "actionId": "act_abc123",
      "senderId": 42,
      "conversationId": "8589934634",
      "messageId": "1001",
      "actionData": "{\"choice\": \"option_a\"}",
      "verb": "submit_form"
    }
  }
}
```

Respond via `MessageService.AnswerCardAction` (toast/alert) or update the card via `MessageService.EditMessage`.

`AnswerCardAction` is `agent_only` ŌĆö called with the Agent's `nxa_` token:

```bash
curl -X POST https://api.nexus-dev.xsyphon.com/api.v1.MessageService/AnswerCardAction \
  -H "Content-Type: application/json" \
  -H "Connect-Protocol-Version: 1" \
  -H "Authorization: Bearer ${AGENT_TOKEN}" \
  -d '{"conversationId": "8589934634", "messageId": "1001", "actionId": "act_abc123", "text": "You chose Blue!", "showAlert": false}'
```

| Field | Description |
|-------|-------------|
| `conversationId` | From the `cardAction` event |
| `messageId` | From the `cardAction` event |
| `actionId` | From the `cardAction` event |
| `text` | Toast or alert text shown to the user |
| `showAlert` | `true` = modal alert, `false` = toast notification |

The user sees the response as a brief toast or an alert dialog. Card actions expire after 5 minutes (TTL).

### CONTACT_ADDED

User added the Agent as a contact. Good place to send a welcome message. This is a sequenced update ŌĆö check for the `snUpdate.contactAdded` field.

```json
{
  "users": [
    {"userId": 42, "nickname": "Alice"}
  ],
  "snUpdate": {
    "sn": 2,
    "contactAdded": {"peerUserId": 42}
  }
}
```

### REMOVED_FROM_GROUP

Agent was removed from a group by the owner. This is a sequenced update ŌĆö check for the `snUpdate.removedFromGroup` field.

```json
{
  "snUpdate": {
    "sn": 3,
    "removedFromGroup": {"groupId": 100, "operatorId": 1}
  }
}
```

### GROUP_DISSOLVED

A group the Agent belongs to was dissolved. This is a sequenced update ŌĆö check for the `snUpdate.groupDissolved` field.

```json
{
  "snUpdate": {
    "sn": 4,
    "groupDissolved": {"groupId": 100, "operatorId": 1}
  }
}
```

## Response Requirements

- Return HTTP 2xx to acknowledge
- Non-2xx responses trigger retries (up to 3 times with exponential backoff)
- Response body content is ignored
- Respond within 5 seconds; timeouts are treated as failures
